# Yealink SIP-T33G Configuration Renderer

**Device Type ID:** `YealinkSIPT33G`  
**Manufacturer:** Yealink  
**Model:** SIP-T33G  
**Max Lines:** 4  
**Config Format:** Plain text (key=value pairs)

## Overview

This document describes all configuration parameters generated by the Yealink SIP-T33G renderer, their sources, and default values. Configuration is grouped by functional area.

## Date & Time

| Configuration Parameter | Value | Source | Notes |
|------------------------|-------|--------|-------|
| `local_time.time_zone` | `+10`, `-5`, etc. | Site: `timezone` | Standard time UTC offset (e.g., +10 for UTC+10) |
| `local_time.summer_time` | `0`, `1`, or `2` | Site: `timezone` | 0=no DST, 1=uses DST, 2=auto-detect DST |
| `local_time.dst_time_type` | `0` or `1` | Site: `timezone` | 0=DST by specific date (not used), 1=DST by day-of-week |
| `local_time.start_time` | `10/1/7/2` | Site: `timezone` | DST start: month/week/day-of-week/hour (Oct 1st Sunday 2am) |
| `local_time.end_time` | `4/1/7/3` | Site: `timezone` | DST end: month/week/day-of-week/hour (Apr 1st Sunday 3am) |
| `local_time.offset_time` | `60` or `0` | Site: `timezone` | DST offset in minutes (60 = +1 hour) |
| `local_time.ntp_server1` | From site | Site: `primary_ntp_ip` | Primary NTP server IP |
| `local_time.ntp_server2` | From site | Site: `secondary_ntp_ip` | Secondary NTP server IP |
| `local_time.interval` | `1440` | Hard-coded | NTP sync interval (minutes, 24h) |
| `local_time.date_format` | `5` (DD MMM YYYY) | Device Option: `local_date_format` | Date display format (0-6) |
| `local_time.time_format` | `1` (24 Hour) | Device Option: `local_time_format` | Time display format (0=12h, 1=24h) |
| `local_time.manual_ntp_srv_prior` | `1` | Hard-coded | Use manual NTP servers (1=yes, 0=no) |

**Date Format Mapping:**
- `0` = WWW MMM DD
- `1` = DD-MMM-YY
- `2` = YYYY-MM-DD
- `3` = DD/MM/YYYY
- `4` = MM/DD/YY
- `5` = DD MMM YYYY (default)
- `6` = WWW DD MMM

**Time Transition Format:** `month/week/day-of-week/hour`
- `month`: 1-12 (1=January, 10=October, etc.)
- `week`: 1-5 (1=first occurrence, 2=second, etc.)
- `day-of-week`: 7=Sunday, 1=Monday, 2=Tuesday, ..., 6=Saturday
- `hour`: 0-23 (24-hour format)

**Examples:**
- `10/1/7/2` = First Sunday in October at 2am (Australia DST starts)
- `4/1/7/3` = First Sunday in April at 3am (Australia DST ends)
- `3/2/7/2` = Second Sunday in March at 2am (US/EU DST starts)
- `11/1/7/2` = First Sunday in November at 2am (US/EU DST ends)

**Default Codec Order:** PCMA, PCMU, G722

## Timezone & DST Logic

The renderer automatically calculates the correct UTC offset and DST transition dates based on the site's IANA timezone name:

1. **Standard Offset**: Samples Jan 15 and Jul 15 to determine which is standard time (smaller offset)
2. **DST Detection**: Checks if offsets differ between January and July
3. **Transition Dates**: Uses pytz internal transition data to find exact DST switch dates
4. **Fallback**: Uses common hemisphere patterns if transition data unavailable

**Output Format:**
- `time_zone`: Integer with +/- sign (e.g., `+10`, `-5`, `+5:30`)
- `summer_time`: 0=no DST, 1=timezone uses DST, 2=auto-detect (Yealink will determine)
- `dst_time_type`: 1=DST by day-of-week (only supported format)
- `start_time` & `end_time`: DST transitions as `month/week/dow/hour`
- `offset_time`: DST offset in minutes (typically 60 for +1 hour)

**Hemisphere Patterns (if transition data unavailable):**
- **Northern Hemisphere**: DST starts 2nd Sunday in March, ends 1st Sunday in November
- **Southern Hemisphere**: DST starts 1st Sunday in October, ends 1st Sunday in April

**Example (Australia/Sydney on Jan 22):**
- `time_zone = +10` (standard time UTC+10)
- `summer_time = 1` (timezone uses DST)
- `dst_time_type = 1` (DST by day-of-week)
- `start_time = 10/1/7/2` (1st Sunday October at 2am)
- `end_time = 4/1/7/3` (1st Sunday April at 3am)
- `offset_time = 60` (add 60 minutes for DST = UTC+11)

## Implementation Notes

- **Encryption**: Line passwords are encrypted in the database and decrypted at render time
- **Content Type**: Configurations are rendered as `text/plain` (not XML)
- **Line Limit**: Maximum 4 lines per device; additional lines are ignored
- **Account Disabling**: Unused account slots (5-4 if only 2 lines) are explicitly disabled
- **MAC Normalization**: MAC addresses are normalized to lowercase with colons
- **Deterministic**: Same input always produces identical output (no random values)
- **Stateless**: No local file storage; all data from database
